Structural Properties & Invariants

AVL Tree

  • Binary search tree ordering at every node
  • Balance factor = height(left) − height(right)
  • |balance factor| ≤ 1 at every node (strict)
  • Rebalanced via single or double rotations
  • Strictest balance → shortest height → fastest lookup
  • More rotations on insert/delete than Red-Black

Red-Black Tree

  • Every node is RED or BLACK
  • Root is always BLACK
  • Every NIL leaf is BLACK
  • RED node → both children must be BLACK
  • All root-to-leaf paths have equal black-height
  • At most 2 rotations per insert, 3 per delete

Binary Heap

  • Complete binary tree stored as a flat array
  • Min-heap: parent ≤ both children at every node
  • Parent of index i is at ⌊(i−1)/2⌋
  • Children of i are at 2i+1 and 2i+2
  • No ordering between siblings
  • Heapify (build) runs in O(n) — not O(n log n)

Binomial Heap

  • Forest of binomial trees Bₖ (one per bit of n)
  • Bₖ has 2ᵏ nodes and height k
  • At most ⌊log₂ n⌋ + 1 trees in the forest
  • Each tree satisfies min-heap ordering
  • Merge is O(log n) — key advantage over Binary Heap
  • Supports Decrease-Key in O(log n)

B+ Tree (T = 3)

  • All data stored only in leaf nodes
  • Internal nodes hold separator keys only
  • Every non-root node has ≥ T−1 = 2 keys
  • Every node has ≤ 2T−1 = 5 keys
  • All leaves at the same depth (perfectly balanced)
  • Leaves linked in a doubly-linked list for range scans
  • Leaf split: copy-up (key stays in right leaf)
  • Internal split: push-up (middle key removed from child)
When to Use Each Structure

AVL Tree

✔ Read-heavy workloads
✔ Fastest lookup needed
✔ Ordered traversal
✔ Predecessor / successor
✘ Frequent inserts/deletes
✘ Heap-merge operations

Red-Black Tree

✔ Balanced read/write mix
✔ General-purpose BST
✔ Used in std::map / TreeMap
✔ Fewer rotations than AVL
✘ Slightly taller than AVL
✘ Heap-merge operations

Binary Heap

✔ Simple priority queue
✔ O(1) find-min
✔ Heapsort (in-place)
✔ Cache-friendly array
✘ No efficient merge
✘ No ordered search

Binomial Heap

✔ Mergeable priority queue
✔ O(log n) merge
✔ Decrease-Key support
✔ Dijkstra / Prim's MST
✘ Slower find-min than BHeap
✘ Pointer overhead

B+ Tree

✔ Database / disk storage
✔ Range queries (O(log n+k))
✔ Sequential scans via leaf list
✔ High fan-out reduces I/O
✘ Complex split/merge logic
✘ No O(1) find-min
